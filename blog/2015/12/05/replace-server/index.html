月間1000万PV超あるサービスのサーバリプレースのマネージメントをした話

Dec 5th, 2015

自分が投稿したQiitaからの転載です

はじめに

基本的にプレイヤー側ではありません。

主に以下のことをしていました

  • 作業の進捗管理
  • 作業内で発生する諸技術の調整
  • 技術サポート

リプレース対象の月間1000万PV超サイトの規模感としては、以下のものです

  • 月間1000万PV超
  • ユーザー数30万人超
  • オンプレミス環境(コストを掛けて物理で殴る感)
  • 移設前のサーバー台数は6台

背景

リプレース対象のサービスは元々、友人が携わているサービスでした。

自分は全く関わりが無いサービスでしたが、友人から困っているとの相談を受けました、

話を掻い摘んで要約すると以下の模様

  1. 「2015年末にデータセンターが統合されるので、年末までの統合先のデータセンターに移設するか、利用停止するかの命令が来た」のでどうにかしたい
  2. とはいえサーバ運用やOSレベルでそこそこわかる人間がいないので、年末までに完了させるのは難しい
  3. 元々、サーバ構築した人間がいないので、ブラックボックス化しているのでなんとかしたい
  4. 動作環境がかなり複雑化しているので、うまく解体してスピード感ある形にしたい
  5. 元々、データセンターのオンプレミス環境には不満があって、クラウドサーバーへの移設を検討していた

1~3の事情があってヤバイけど、4,5も進めていきたいという模様でしょうか

移設前後の構成

後から、尾を引きたくなるようになることは間違い無かったので
サーバを役割ごとに分割しました

移設前

全台オンプレミス環境で以下の構成で運用されていました

  • Webサーバ + その他複数の機能兼用サーバ1台
  • Webサーバ2台
  • Master/Slave構成のデータベースサーバが各1台ずつ
  • Image(画像配信)サーバ1台

移設後

全台クラウド環境で、以下の構成に変更しました

  • Webサーバ3台
  • Batchサーバ1台
  • Master/Slave構成のデータベースサーバが各1台ずつ
  • Image(画像配信)サーバ1台
  • Mail(メール配信)サーバ1台

その他複数機能あったサーバをうまくRoleごとに切り分けることができました

マネージメントとしてやったこと

基本的に作業は任せる形にして、以下の方向性だけを決めるのに留めました (ぶっちゃけ、細かいサービスレベルの話や稼働しているアレコレの内容が不明な為)

  • 移設方針の決定
  • マイルストーンの策定
  • 現行サービス構成の再確認と共有
  • 作業ルールの策定
  • 進捗管理と各種作業の調整
  • タスク管理
  • 技術サポート

ぐらいですかね。これだけ見ると、殆ど口だけだしてタスク状況を把握しているだけの感じですねｗ

作業に関しては、基本的にと言いましたが、進捗状況で一部を実施する形にはなりました。 (サービスに影響が全く無いわけでは無いけれども、一般的な手順で解決できるようなこと)

まとめ
